# Monorepo + Orval Integration Skill

You are integrating a standalone NestJS 11 backend and a React 19 frontend into a pnpm monorepo with a typed HTTP client generated by Orval. This workflow has been validated end-to-end — follow each phase in order and **stop for user confirmation after every phase** before continuing.

## Architecture — understand before starting

The goal is this data flow:

```
Component / TanStack Query hooks
    ↓  ($Entity — Zod front types — single source of truth)
services/$entity.service.ts   ← adapter: calls Orval + Zod.parse()
    ↓
services/generated/$entities.ts  ← generated by Orval (internal to services/ only)
    ↓
Backend REST API (NestJS 11)
```

**The frontend owns its types.** Zod schemas in `features/*/schemas/` define the front-side shape. Orval-generated types in `src/api/model/` are internal to `src/api/services/` and never imported by components or queries.

## Phase 1 — pnpm monorepo setup

**Goal:** create workspace structure without touching functional code.

### 0. Root `/.gitignore`

Invoke the `gitignore` skill with arg `pnpm-monorepo`. It writes a single root file and instructs you to delete per-package `.gitignore` files — git patterns apply recursively so one file covers both packages.

### 1. `/pnpm-workspace.yaml`
```yaml
packages:
  - "$FRONTEND_DIR"
  - "$BACKEND_DIR"
```

### 2. Root `/package.json`
```json
{
  "name": "$FRONTEND_PACKAGE_NAME-monorepo",
  "private": true,
  "pnpm": {
    "onlyBuiltDependencies": [
      "@nestjs/core", "@prisma/client", "@prisma/engines",
      "@swc/core", "esbuild", "msw", "prisma"
    ]
  },
  "scripts": {
    "dev": "pnpm --parallel --filter ./$FRONTEND_DIR --filter ./$BACKEND_DIR run dev:package",
    "build": "pnpm --filter ./$BACKEND_DIR run build && pnpm --filter ./$FRONTEND_DIR run build",
    "lint": "pnpm --recursive run lint",
    "test": "pnpm --recursive run test",
    "generate:api": "pnpm --filter ./$BACKEND_DIR run openapi:export && pnpm --filter ./$FRONTEND_DIR run generate:api",
    "db:up": "docker compose -f $BACKEND_DIR/docker-compose.yml up -d",
    "db:migrate": "pnpm --filter ./$BACKEND_DIR run prisma:migrate",
    "db:seed": "pnpm --filter ./$BACKEND_DIR run prisma:seed"
  }
}
```

After creating the root `package.json`, identify any runtime dependency that appears in **both** sub-packages (e.g., `zod`, `axios`). Pin it to a single version via `pnpm.overrides` to prevent version drift and duplicate lockfile entries:

```json
"pnpm": {
  "overrides": {
    "zod": "3.24.2"
  }
}
```

This ensures one install, one lockfile entry, and identical runtime behavior across packages.

### 3. Add `dev:package` scripts to sub-packages
- `$FRONTEND_DIR/package.json`: `"dev:package": "vite"`
- `$BACKEND_DIR/package.json`: `"dev:package": "nest start --watch"`

### 4. Root `/biome.json` (shared linting config)
```json
{
  "$schema": "https://biomejs.dev/schemas/2.4.8/schema.json",
  "formatter": { "enabled": true, "indentStyle": "space", "indentWidth": 2 },
  "linter": { "enabled": true, "rules": { "recommended": true } },
  "assist": { "enabled": true, "actions": { "source": { "organizeImports": "on" } } }
}
```

In each sub-package's `biome.json`, add as first key: `"extends": ["../biome.json"]`

Each sub-package's `lint` and `format` scripts **must** pass `--config-path ..` so Biome resolves the root config when invoked from within the sub-package directory. Biome 2.x does not auto-resolve parent-directory configs from a subdirectory execution context — without this flag, Biome falls back to built-in defaults and silently ignores `../biome.json`.

Update the `lint` and `format` entries in both sub-package `package.json` scripts to:
```json
"lint": "biome lint --config-path .. ./src",
"format": "biome format --config-path .. --write ./src"
```

⚠️ **Do NOT add `@biomejs/biome` to sub-package `devDependencies`.** pnpm hoists the binary from the root installation — sub-packages resolve it via `node_modules/.bin/biome`. Adding it again in a sub-package creates a duplicate lockfile entry with no benefit. `devboard-back` is proof: it has no Biome devDependency yet its `format` script works fine.

### 5. Move pnpm config to root
Remove `pnpm.onlyBuiltDependencies` and `pnpm.overrides` from sub-package `package.json` files if present — they belong only in the root.

### 6. ⚠️ Critical — pnpm workspace strict mode
pnpm does not hoist transitive dependencies. Every package used directly in code **must be an explicit direct dependency**, even if it's already a transitive dep.

Common case: `express` used directly in `$BACKEND_DIR/src/main.ts` via named imports must be listed in `$BACKEND_DIR/package.json` dependencies, even though it's already a dep of `@nestjs/platform-express`.

**Verify:** `pnpm install` from root completes without unresolved import errors.

### Verification
```bash
pnpm --filter ./$FRONTEND_DIR run dev:package   # frontend starts
pnpm --filter ./$BACKEND_DIR run dev:package    # backend starts
```

## Phase 3 — Backend OpenAPI enrichment

**Goal:** produce a clean OpenAPI spec that Orval can consume with meaningful function names and typed responses.

### operationId on every endpoint
```typescript
@ApiOperation({ operationId: 'get$Entities', summary: 'List all $entities' })
```

Naming convention: `get$Entities`, `create$Entity`, `get$EntityById`, `update$Entity`, `delete$Entity`, `get$EntityRelation`

### Typed responses (critical for Orval)
Use `type:` not `schema: { example: ... }` — Orval generates `unknown` when it can't see a type:

```typescript
// ✅ Orval generates: orvalMutator<$EntityResponseDto>
@ApiResponse({ status: 201, type: $EntityResponseDto })
@ApiOkResponse({ type: $EntityResponseDto })

// ❌ Orval generates: orvalMutator<unknown>
@ApiResponse({ status: 201, schema: { example: ENTITY_EXAMPLE } })
```

### Regenerate spec
```bash
pnpm --filter ./$BACKEND_DIR run openapi:export
```

### Verification
```bash
grep "operationId:" $BACKEND_DIR/openapi.yaml   # all endpoints listed
grep "^\s*{}" $BACKEND_DIR/openapi.yaml          # should return nothing
```

## Phase 4 — Orval installation and configuration

### Install
```bash
pnpm --filter ./$FRONTEND_DIR add -D orval
```

### Create `$FRONTEND_DIR/src/api/client/orval-mutator.ts`
```typescript
import type { AxiosRequestConfig, AxiosResponse } from "axios";
import { apiClient } from "./axios.client";

export const orvalMutator = <T>(config: AxiosRequestConfig): Promise<AxiosResponse<T>> => {
  return apiClient(config);
};

export type ErrorType<Error> = import("axios").AxiosError<Error>;
export type BodyType<BodyData> = BodyData;
```

### Create `$FRONTEND_DIR/orval.config.ts`
```typescript
import { defineConfig } from "orval";

export default defineConfig({
  $FRONTEND_PACKAGE_NAME: {
    input: {
      target: "../$BACKEND_DIR/openapi.yaml",
    },
    output: {
      mode: "tags",                          // 1 file per OpenAPI tag → $entities.ts
      target: "src/api/services/generated",  // internal — never import outside services/
      schemas: "src/api/model",              // internal types — never import outside services/
      client: "axios",
      mock: false,
      override: {
        mutator: {
          path: "./src/api/client/orval-mutator.ts",
          name: "orvalMutator",
        },
      },
    },
  },
});
```

### Add script to `$FRONTEND_DIR/package.json`
```json
"generate:api": "orval --config orval.config.ts"
```

### Run generation
```bash
pnpm --filter ./$FRONTEND_DIR run generate:api
```

### ⚠️ Critical — Orval mode:tags factory pattern
With `mode: "tags"` + `client: "axios"`, Orval generates a **factory function**, not direct exports:

```typescript
// ✅ Correct usage
import { get$Entities as get$EntitiesClient } from "@/api/services/generated/$entities";
const client = get$EntitiesClient();
await client.get$Entities();
await client.create$Entity(dto);

// ❌ Wrong — get$Entities is a factory, not the HTTP function itself
import { get$Entities } from "@/api/services/generated/$entities";
await get$Entities();
```

### Verification
Check generated file types — should not contain `unknown`:
```bash
grep "orvalMutator<" $FRONTEND_DIR/src/api/services/generated/$entities.ts
# Expected: orvalMutator<$EntityResponseDto[]>, orvalMutator<$EntityResponseDto>, orvalMutator<void>
# Not: orvalMutator<unknown>
```

## Phase 5 — Services adapter layer

**Goal:** make existing services call Orval functions instead of raw axios, while keeping the same public interface (types, method names).

Components and TanStack Query hooks are **not changed** — they continue using `$entityService` and front Zod types.

### Pattern — `$FRONTEND_DIR/src/api/services/$entity.service.ts`
```typescript
import type { Create$EntityInput, $Entity, Update$EntityInput } from "@/features/$entities/schemas/$entity.schema";
import { $entitySchema, $entitiesResponseSchema } from "@/features/$entities/schemas/$entity.schema";
import { get$Entities as get$EntitiesClient } from "@/api/services/generated/$entities";

const client = get$EntitiesClient();

export const $entityService = {
  getAll: async (): Promise<$Entity[]> => {
    const { data } = await client.get$Entities();
    return $entitiesResponseSchema.parse(data);  // validates + types as front $Entity[]
  },
  create: async (input: Create$EntityInput): Promise<$Entity> => {
    const { data } = await client.create$Entity(input);
    return $entitySchema.parse(data);
  },
  update: async (id: string, input: Update$EntityInput): Promise<$Entity> => {
    const { data } = await client.update$Entity(id, input);
    return $entitySchema.parse(data);
  },
  delete: async (id: string): Promise<void> => {
    await client.delete$Entity(id);
  },
};
```

Apply the same pattern for each entity in the app.


## Known pitfalls — read before debugging

### 1. NestJS + Express 5 double body parser → 500 on all POST/PATCH ⭐ Most common

**Symptom:** all write operations return `500 "stream is not readable"`, GET works fine.

**Cause:** `NestFactory.create()` enables a built-in body parser by default. Adding `express.json()` manually adds a second one that consumes the request stream first. NestJS then tries to read an already-consumed stream.

**Fix:**
```typescript
import * as bodyParser from 'body-parser';

// Disable NestJS built-in, register body-parser explicitly
const app = await NestFactory.create(AppModule, { bodyParser: false });
app.use(bodyParser.json({ limit: '100kb' }));
app.use(bodyParser.urlencoded({ limit: '100kb', extended: true }));
```

Add as direct dependencies:
```bash
pnpm --filter ./$BACKEND_DIR add body-parser
pnpm --filter ./$BACKEND_DIR add -D @types/body-parser
```

---

### 2. Prisma enums vs string values → 500 on create/update

**Symptom:** 500 when creating or updating an entity with an enum field.

**Cause:** Prisma 6 expects the typed enum (`$EntityStatus.active`), not the plain string `"active"`. Zod validates the string correctly, but passing it directly to Prisma fails.

**Fix** (in service):
```typescript
import { $EntityStatus } from '@prisma/client';

// create
data: { ...dto, status: dto.status as $EntityStatus }

// update
data: { ...dto, status: dto.status ? (dto.status as $EntityStatus) : undefined }
```

---

### 3. Orval returns `unknown` on create/update

**Symptom:** `orvalMutator<unknown>` in generated files instead of `orvalMutator<$EntityResponseDto>`.

**Cause:** `@ApiResponse({ schema: { example: ... } })` doesn't expose type info to Orval.

**Fix:**
```typescript
// ✅
@ApiResponse({ status: 201, type: $EntityResponseDto })
@ApiOkResponse({ type: $EntityResponseDto })

// ❌
@ApiResponse({ status: 201, schema: { example: ENTITY_EXAMPLE } })
```

---

### 4. pnpm workspace strict mode — transitive deps not accessible

**Symptom:** `Cannot find module 'express'` at runtime, even though express is installed.

**Cause:** pnpm strict mode — packages must be explicit direct dependencies to be importable.

**Fix:** add the package explicitly:
```bash
pnpm --filter ./$BACKEND_DIR add express
```

---

### 5. TypeScript TS2589 deep instantiation with zodResolver

**Symptom:** `Type instantiation is excessively deep and possibly infinite` in edit form components.

**Cause:** Zod schemas with `.default()` fields trigger deep type inference via React Hook Form + `zodResolver`.

**Fix:**
```typescript
// biome-ignore lint/suspicious/noExplicitAny: prevents TS2589 deep instantiation with Zod default fields
resolver: zodResolver(update$EntitySchema as any),
```

---

### 6. supertest import with Vitest + unplugin-swc

**Symptom:** `TypeError: (0, __vite_ssr_importX__) is not a function` on every `request(app.getHttpServer())` call in e2e tests.

**Cause:** With the `unplugin-swc` transformer used for e2e tests, `import * as request from 'supertest'` gives the module namespace object, not the callable default export.

**Fix:** Use the default import form:
```typescript
import request from 'supertest';   // ✅ callable default export
// NOT:
import * as request from 'supertest';  // ❌ namespace object — not callable
```
